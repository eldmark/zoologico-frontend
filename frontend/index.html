Factory Method

El cliente solo llama a crearAnimal(dto). La fábrica decide qué subclase concreta instanciar —Leon, Tiburon, Aguila— sin que el resto del código lo sepa.

AnimalFactory (abstract)
└── AnimalFactoryImpl
├── new Leon(...)
├── new Tiburon(...)
└── new Aguila(...)
Bridge

Animal y Hábitat son jerarquías independientes unidas por composición. Sin Bridge necesitarías clases como LeonSabana, TiburonRio… Con Bridge, cualquier combinación válida es libre.

Animal ──uses──▶ Habitat (interface)
├── Sabana
├── Rio
└── Montana
Chain of Responsibility

Cada eslabón valida una regla y pasa al siguiente. Si falla, corta la cadena. El último eslabón tiene la regla de negocio real: ¿puede este animal vivir en este hábitat?

ValidarNombre
→ ValidarEdad
→ ValidarTipo
→ ValidarHabitat
→ ValidarCompatibilidad ★
DTO — Data Transfer Object

El formulario empaqueta los datos en un objeto plano antes de enviarlo al servidor. El DTO no tiene lógica, solo transporta.

// Lo que viaja en el POST: { "nombre": "Simba", "edad": 5, "tipo": "LEON", "habitat": "SABANA" } // En Java: class AnimalDTO { String nombre, tipo, habitat; int edad; }
Cliente–Servidor

El frontend (HTML/JS) y el backend (Spring Boot) son procesos separados. El cliente envía peticiones HTTP y el servidor responde con JSON. Nunca comparten memoria.

Cliente (puerto :5500) POST /api/animales → JSON GET /api/animales → [ ] GET /api/animales/ordenar Servidor (puerto :8080) @RestController @CrossOrigin(origins = "*")
